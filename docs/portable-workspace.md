# Nova Portable Workspace

A **Nova Portable Workspace** is an optional way to run Nova: instead
of letting the Git checkout, the SQLite database, the backups, and
the config drift across `/home`, `/var`, and `/etc`, you keep
everything under one parent folder. The whole thing — code, data,
config, logs, backups — can then be moved between disks or machines
as a single unit.

This document explains the layout, how to scaffold a workspace, how
to wire it to **systemd** *or* **Docker**, how to back it up, and how
to restore it.

The workspace is **opt-in**. Existing installs keep working unchanged.
Nothing in this layout is required by Nova itself; it is purely a
convention on top of the `NOVA_DATA_DIR` foundation documented in
[`docs/data-directory.md`](data-directory.md).

## What a portable workspace is

A directory tree like this:

```
NovaPortable/
├── README.md             # generated by the init helper (short pointer)
├── app/
│   └── Nova/             # Git checkout (you clone this manually)
├── data/                 # NOVA_DATA_DIR points here
│   ├── nova.db
│   ├── nova.db.backup
│   ├── backups/
│   ├── exports/
│   ├── memory-packs/
│   └── logs/
├── logs/                 # workspace-level logs (future use)
├── backups/              # operator-owned backup packs
├── config/
│   ├── nova.env.example  # generated by the init helper
│   └── nova.env          # you create this; systemd reads it
└── scripts/              # your own start/stop/backup helpers
```

The architectural rule is simple:

* `app/Nova/` holds the **Git checkout**. Public code. Safe to push,
  pull, diff, and `rm -rf`.
* `data/` holds **private runtime data**. Conversations, memories,
  feedback, user settings. Never committed to Git.
* `config/` holds **local configuration**, including secrets like
  `NOVA_PASSWORD` and `NOVA_SECRET_KEY`. Never committed to Git.
* `logs/` and `backups/` are operator-owned; Nova does not touch them
  unless you point it there.
* `scripts/` is yours — drop in `start.sh`, `stop.sh`, `backup.sh`,
  `status.sh`, or whatever you like. The Nova repo does not ship
  any of these because their right contents depend on your host
  (Docker vs. systemd vs. bare process).

## Why this is safer than storing data inside the repo

* **Private data does not leak into Git.** A `git clean -xdf` will
  not erase `nova.db` because the database lives outside `app/Nova/`.
  A misconfigured `.gitignore` cannot accidentally stage it either.
* **Backing up is one folder.** You snapshot the parent, not five
  scattered paths.
* **Moving disks is `rsync`, not a migration plan.** Stop Nova,
  `rsync -a` the parent to the new mount, update systemd's
  `EnvironmentFile=` (and `WorkingDirectory=`), restart.
* **Two Nova instances on one host don't fight.** Each one owns its
  workspace folder, including its own `nova.db` and its own config.
* **`git pull` is still a no-op for runtime state.** Code updates
  affect `app/Nova/` only; your data is in `data/`.

## Creating a workspace

The Python helper that ships with Nova creates the empty directory
layout and a single example env file. It does **not** clone Nova,
copy `nova.db`, or move anything off your existing install.

### Option A — Python module (recommended)

From an existing Nova checkout:

```bash
python -m core.paths init-workspace /mnt/fastdata/NovaPortable
```

### Option B — Shell wrapper

```bash
scripts/init-portable-workspace.sh /mnt/fastdata/NovaPortable
```

Either invocation creates:

```
/mnt/fastdata/NovaPortable/
├── README.md
├── app/
├── data/
├── logs/
├── backups/
├── config/
│   └── nova.env.example
└── scripts/
```

The helper is **idempotent and safe to re-run**:

* If the parent already exists, it is reused.
* If any subdirectory already exists, it is left untouched (contents
  are not touched either).
* If `config/nova.env.example` already exists, it is **not**
  overwritten — the helper keeps your local copy.
* If the workspace-level `README.md` already exists, it is **not**
  overwritten either — the helper keeps your local copy.
* If a name already exists but is not a directory, the helper exits
  with a clear error naming the offending path.

The generated `README.md` is a short pointer file: it names what each
subdirectory is for, restates the "data stays out of Git" rule, and
points back at this document. If you maintain your own site-specific
runbook at the workspace root, drop it in as `README.md` before
running the helper and your copy will be preserved.

The helper does not write into `data/` — that is reserved for Nova at
runtime — and it does not write into `app/`. The Git checkout is your
job (see below).

### Clone Nova into `app/Nova`

```bash
git clone https://github.com/thezupzup/nova.git \
          /mnt/fastdata/NovaPortable/app/Nova
cd /mnt/fastdata/NovaPortable/app/Nova
python3 -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
```

Nothing forces the checkout to live at `app/Nova`. If you prefer
`app/nova-stable` and `app/nova-beta` for two channels, that is fine —
adjust systemd's `WorkingDirectory=` accordingly.

### Configure `config/nova.env`

```bash
cp /mnt/fastdata/NovaPortable/config/nova.env.example \
   /mnt/fastdata/NovaPortable/config/nova.env
chmod 600     /mnt/fastdata/NovaPortable/config/nova.env
$EDITOR       /mnt/fastdata/NovaPortable/config/nova.env
```

The generated example file contains:

```env
NOVA_DATA_DIR=/mnt/fastdata/NovaPortable/data
```

Add the rest of the variables you need (`NOVA_USERNAME`,
`NOVA_PASSWORD`, `NOVA_SECRET_KEY`, `OLLAMA_HOST`, …). Anything that
fits in `.env.example` at the top of the repo also fits in
`config/nova.env`. Keep this file private — `chmod 600` is a good
default.

## Picking a disk for the workspace

The portable layout makes it easy to put the workspace on whichever
disk you want, and to split active data from archives. The split that
works well in practice:

| Disk        | Holds                                          | Why                          |
|-------------|------------------------------------------------|------------------------------|
| SSD (NVMe)  | the live workspace — `app/`, `data/`, `config/`| SQLite writes are random I/O |
| HDD or NAS  | `backups/` snapshots, archived exports         | sequential reads, cheaper GB |

Put the **active** workspace on an SSD if you can. Nova writes to
`nova.db` on every message — conversation rows, memory rows, feedback
rows, settings — and SQLite's per-statement `fsync` turns those into
random writes. On a spinning disk this shows up as a perceptible lag
between hitting *Send* and the assistant bubble starting to stream;
on NVMe it is invisible. The same disk hosts `memory-packs/` and
`exports/`, which are small but read-heavy when the admin surface
lists them.

Backups (snapshots, Phase 2 export tarballs, off-host archives) are
ideally on a **separate** disk, and **slower storage is fine** for
them. Backups are written sequentially, read rarely, and you want
them on different hardware from the live database anyway — if the
SSD fails, an archive that lives on the same drive fails with it.
A spinning HDD, a NAS share, or an external USB disk attached to a
stable mount point all work. The portable layout supports this split
naturally: the live workspace lives at `/mnt/fastdata/NovaPortable/`,
and `rsync` snapshots land in `/mnt/archive/NovaPortable-<stamp>/`
on a slower disk.

What you should avoid:

* **Live `data/` on a spinning USB disk.** SQLite's commit pattern is
  brutal on slow random I/O and can stretch a single chat round-trip
  from milliseconds to seconds.
* **Live `data/` on a network share.** NFS, SMB, and SSHFS do not
  give SQLite the locking guarantees it expects. Use them for
  `backups/` instead, never for `data/`.
* **`/run/media/<user>/<disk>`** — see "Why stable mount paths
  matter" below.

## Picking a deployment shape

Nova does not care whether it runs as a systemd service or inside a
container. The portable workspace works for both, but the wiring is
slightly different:

| Shape   | Best for                                  | Reads `nova.env` via   |
|---------|-------------------------------------------|------------------------|
| systemd | bare-metal hosts, NAS appliances, VMs     | `EnvironmentFile=`     |
| Docker  | disposable containers, multi-host moves   | `env_file:` + bind mounts |

The directory layout under `NovaPortable/` is identical in both
cases. The only difference is **where Nova reads its data from** —
the host path on systemd, or the in-container `/data` path on Docker.

## Wiring the workspace to systemd

Systemd reads the workspace's env file via `EnvironmentFile=`. A
minimal unit for the portable layout looks like this:

```ini
[Unit]
Description=Nova AI Assistant (portable workspace)
After=network.target ollama.service
RequiresMountsFor=/mnt/fastdata

[Service]
Type=simple
User=nova
Group=nova
EnvironmentFile=/mnt/fastdata/NovaPortable/config/nova.env
WorkingDirectory=/mnt/fastdata/NovaPortable/app/Nova
ExecStart=/mnt/fastdata/NovaPortable/app/Nova/.venv/bin/python3 \
          -m uvicorn web:app --host 0.0.0.0 --port 8080
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

The `RequiresMountsFor=` line tells systemd to wait for the
`/mnt/fastdata` mount before starting Nova and to stop the unit
cleanly if the disk is unmounted. Use the **stable mount path** from
`/etc/fstab` — see the note below.

The hardened example unit shipped at
[`deploy/systemd/nova.service`](../deploy/systemd/nova.service)
applies the full sandbox profile (read-only filesystem, dropped
capabilities, syscall denylist, etc.). To use it with the portable
workspace, after the placeholder substitutions:

```ini
[Service]
EnvironmentFile=/mnt/fastdata/NovaPortable/config/nova.env
WorkingDirectory=/mnt/fastdata/NovaPortable/app/Nova
ExecStart=/mnt/fastdata/NovaPortable/app/Nova/.venv/bin/uvicorn \
          web:app --host 0.0.0.0 --port 8080
ReadWritePaths=/mnt/fastdata/NovaPortable/app/Nova
ReadWritePaths=/mnt/fastdata/NovaPortable/data
ReadWritePaths=/mnt/fastdata/NovaPortable/logs
ReadWritePaths=/mnt/fastdata/NovaPortable/backups
```

`ReadWritePaths=` repeats are additive in systemd — keep one path per
line so the diff is easy to review. The `config/` directory does not
need to be writable.

Install and start:

```bash
sudo cp /path/to/nova.service /etc/systemd/system/nova.service
sudo systemctl daemon-reload
sudo systemctl enable --now nova
journalctl -u nova -n 50 --no-pager
```

## Wiring the workspace to Docker / container deployments

The portable workspace is a natural fit for Docker because the
container becomes **disposable** — the moment all of Nova's state
lives in bind-mounted host directories, you can rebuild or replace
the container without losing memory, settings, or backups.

### Container path conventions

Inside the container, Nova always reads its data from a single
well-known path:

| Container path | Purpose                                | Host source |
|----------------|----------------------------------------|-------------|
| `/data`        | `NOVA_DATA_DIR` — nova.db, exports, …  | `NovaPortable/data/` (read/write) |
| `/config`      | Optional read-only config bundle       | `NovaPortable/config/` (read-only) |
| `/logs`        | Optional file logs (future use)        | `NovaPortable/logs/` (read/write) |

`/data` is the contract. Anything else is optional. Keeping
`NOVA_DATA_DIR=/data` constant means the image does not need to know
where on the host the data lives — that decision is moved to the
volume mount and is trivial to change.

### docker-compose example

A ready-to-edit copy lives at
[`deploy/docker/docker-compose.portable.yml`](../deploy/docker/docker-compose.portable.yml).
The relevant fragment looks like this — drop it into a file next to
your workspace (for example, `NovaPortable/docker-compose.yml`) and
adjust the workspace path:

```yaml
services:
  nova:
    image: ghcr.io/thezupzup/nova:latest
    # Or build from a local checkout under app/Nova:
    # build: ./app/Nova
    container_name: nova
    restart: unless-stopped
    env_file:
      - /mnt/fastdata/NovaPortable/config/nova.env
    environment:
      NOVA_DATA_DIR: /data
      OLLAMA_HOST: http://host.docker.internal:11434
    volumes:
      # Live database, backups, exports, memory-packs.
      - /mnt/fastdata/NovaPortable/data:/data
      # Optional: read-only config view, useful for future tooling.
      - /mnt/fastdata/NovaPortable/config:/config:ro
      # Optional: file logs (Nova does not write here by default).
      - /mnt/fastdata/NovaPortable/logs:/logs
    ports:
      - "8080:8080"
    extra_hosts:
      # Lets the container reach an Ollama running on the Docker host
      # on Linux, matching macOS/Windows behaviour. Harmless elsewhere.
      - "host.docker.internal:host-gateway"
```

Then:

```bash
cd /mnt/fastdata/NovaPortable
docker compose up -d
docker compose logs -f nova
```

To update the image without touching data:

```bash
cd /mnt/fastdata/NovaPortable
docker compose pull
docker compose up -d
```

Because `data/` is a bind mount on the host, **deleting or recreating
the container does not delete Nova memory.** A `docker compose down`
(without `-v`) tears down the container but leaves the host directory
untouched. Even `docker compose down -v` is safe with bind mounts —
Docker only removes named volumes, not host paths.

### Why `/data`?

* It is short, conventional, and matches the path Nova's existing
  `docker-compose.yml` and entrypoint use.
* It is independent of the host layout. Moving from
  `/mnt/fastdata/NovaPortable` to `/mnt/newdisk/NovaPortable` only
  requires editing the volume line, not rebuilding the image.
* `/data` is the documented `NOVA_DATA_DIR` value across Docker docs,
  so a single env file works for both compose and bare `docker run`.

### Difference from the named-volume Docker quickstart

The project's existing [`docs/docker.md`](docker.md) uses a Docker
**named volume** (`nova-data`) for the database. That is convenient
for a single-host quickstart but harder to move between machines.
The portable layout swaps the named volume for a **bind mount** on
the host so you can:

* see, snapshot, and `rsync` the data without entering Docker;
* move the workspace to another machine by copying one directory;
* keep the systemd and Docker shapes pointing at the same path.

You do not need to use one or the other — pick whichever fits your
host. The portable layout is just a recipe that works in both
worlds.

### What you should NOT mount

The container is a security boundary, not a magic one. A few mounts
make Nova trivially exploitable and must not be added "for
convenience":

* **Do not mount the Docker socket** (`/var/run/docker.sock`) into
  Nova. Anything that can write to the socket can spawn privileged
  containers and break out to the host.
* **Do not run the container with `--privileged`** or with
  `cap_add: [SYS_ADMIN]`. Nova does not need any extra capability.
* **Do not mount `/`** or the host root filesystem (`/:/host` and
  friends). The portable workspace is exactly three host paths;
  nothing else needs to be visible from inside the container.
* **Do not mount `$HOME`** or the host user's home directory. SSH
  keys, GPG keys, browser profiles, and shell history must stay
  outside Nova.
* **Do not bake tokens into the image**. GitHub, Plex, Jellyfin, or
  any other integration credentials belong in `config/nova.env` on
  the host — readable only by the operator's user — and reach the
  container via `env_file:` at run time. They never go in the
  Dockerfile or in image labels.
* **Do not bind-mount Ollama's model directory** unless you have a
  specific reason. Ollama runs as its own service; Nova talks to it
  over HTTP via `OLLAMA_HOST`, and that's enough.

Docker isolation is a useful **defence in depth** layer, not a
substitute for keeping the rest of your host hardened. The
[hardened systemd unit](../deploy/systemd/nova.service) gives the
bare-metal install a similar reduction in blast radius; the
combination (hardened systemd + container) is fine when you want
both.

### Ollama and other side services

Keep Ollama **outside** the Nova container:

* On Linux, run Ollama as a normal service and bind it to an
  interface the container can reach (`OLLAMA_HOST=0.0.0.0`).
* From inside the container, set
  `OLLAMA_HOST=http://host.docker.internal:11434`. The
  `extra_hosts: ["host.docker.internal:host-gateway"]` entry in the
  compose example wires that up on Linux; macOS / Windows Docker
  Desktop provides the alias natively.

Models are pulled by the **host's** Ollama, not by Nova. Nova never
downloads model weights and never writes into Ollama's data
directory. Keep Ollama models out of the Nova workspace; if you back
the workspace up, you also back up only what Nova owns.

The same rule applies to Plex, Jellyfin, SilentGuard, and any other
side service: they each manage their own data directory. Nova reads
metadata over the network when an integration is enabled, and writes
nothing back outside its own `data/` directory.

## Moving the workspace to another disk

### systemd

```bash
# 1. Stop Nova so the database is not being written to.
sudo systemctl stop nova

# 2. Copy the entire workspace to the new mount. rsync -aH preserves
#    timestamps, permissions, and hard links.
sudo rsync -aH /mnt/fastdata/NovaPortable/ \
               /mnt/newdisk/NovaPortable/

# 3. Update the env file and systemd unit to point at the new path.
sudo $EDITOR /mnt/newdisk/NovaPortable/config/nova.env
sudo $EDITOR /etc/systemd/system/nova.service   # EnvironmentFile=, WorkingDirectory=, ReadWritePaths=
sudo systemctl daemon-reload

# 4. Start Nova from the new location.
sudo systemctl start nova
journalctl -u nova -n 50 --no-pager

# 5. Once you have verified the new instance works end-to-end,
#    the old directory can be archived or deleted manually.
```

### Docker

```bash
# 1. Stop and remove the container (keeps the bind-mounted data).
cd /mnt/fastdata/NovaPortable
docker compose down

# 2. Copy the entire workspace to the new mount.
sudo rsync -aH /mnt/fastdata/NovaPortable/ \
               /mnt/newdisk/NovaPortable/

# 3. Update the volume paths in the compose file (or simply edit the
#    leading directory in each `- <host>:<container>` line).
$EDITOR /mnt/newdisk/NovaPortable/docker-compose.yml

# 4. Bring Nova back up from the new location.
cd /mnt/newdisk/NovaPortable
docker compose up -d
docker compose logs -f nova

# 5. Once verified, remove the old workspace by hand.
```

Nova never moves data on its own. The old workspace stays on disk
until you remove it.

## Backing it up

The portable layout is designed to be backed up as a single tree.
Stop Nova first so the SQLite file is consistent, then snapshot.

### systemd

```bash
sudo systemctl stop nova

sudo -u nova rsync -aH \
    /mnt/fastdata/NovaPortable/ \
    /mnt/archive/NovaPortable-$(date +%Y%m%dT%H%M%SZ)/

sudo systemctl start nova
```

### Docker

```bash
cd /mnt/fastdata/NovaPortable
docker compose stop nova

sudo rsync -aH \
    /mnt/fastdata/NovaPortable/ \
    /mnt/archive/NovaPortable-$(date +%Y%m%dT%H%M%SZ)/

docker compose start nova
```

### Live SQLite snapshot

If a live snapshot is required, use SQLite's own backup API for the
database and `rsync` for everything else:

```bash
sudo -u nova sqlite3 /mnt/fastdata/NovaPortable/data/nova.db \
    ".backup '/mnt/archive/nova-$(date +%Y%m%dT%H%M%SZ).db'"

sudo -u nova rsync -aH --exclude='data/nova.db' \
    /mnt/fastdata/NovaPortable/ \
    /mnt/archive/NovaPortable-$(date +%Y%m%dT%H%M%SZ)/
```

### Portable export packages (Phase 2/3 tools)

`rsync` is the right tool for whole-workspace snapshots on the same
host. When you want a **portable** package — one tarball you can move
to a new machine, hand to a colleague, or hold in cold storage — use
Nova's built-in export tools instead. They produce a single
allowlisted `tar.gz` containing `manifest.json`, a `RESTORE.md`
instruction sheet, and only the canonical Nova data files. Nothing
outside `NOVA_DATA_DIR` ever ends up in the archive — no `.venv`, no
`.git`, no Ollama models, no host secrets.

Build, inspect, and restore packages from the command line:

```bash
# Build a portable export into the workspace's exports/ directory.
python -m core.data_export export

# Validate an export package before doing anything with it.
python -m core.data_export inspect /path/to/nova-export.tar.gz

# See what a restore would do — never writes, never deletes.
python -m core.data_export restore-dry-run /path/to/nova-export.tar.gz

# Perform the restore. Writes a pre-restore backup first; a failed
# restore leaves the current data bit-for-bit identical.
python -m core.data_export restore /path/to/nova-export.tar.gz
```

The same flows are available in the admin overlay under the
**⚇ Storage** tab. Both surfaces require an admin bearer token; both
honour the same safety contract — every restore writes an automatic
pre-restore backup and refuses to proceed if a `nova.db` already
exists at the target unless you confirm. See
[`docs/storage-and-migration.md`](storage-and-migration.md) for the
full walkthrough, including the manifest schema and the conflict
resolution rules.

Why use this on top of `rsync`?

* The package is **portable**: a single file, content-addressable,
  easy to verify and easy to encrypt.
* The package is **allowlisted**: only Nova's own data files are
  included, so you cannot accidentally bundle secrets from
  `config/nova.env`, the host's `~/.ssh/`, or an Ollama model
  directory that drifted into the workspace.
* The restore step writes a **pre-restore backup** before replacing
  any file, so a botched restore is reversible.

**Do not include in backups:**

* `.venv/` under `app/Nova/` — recreate from `requirements.txt`.
* `.git/` under `app/Nova/` if your remote already holds the history.
* **Ollama models** — they live outside the workspace by default;
  re-download with `ollama pull` after restore. Bundling them
  bloats backups by tens of GB and they are reproducible.
* Any media library or Jellyfin / Plex database — those are owned by
  other services and have their own backup story.
* SSH keys, GPG keys, OAuth tokens, or API keys stored elsewhere on
  the host.
* `/tmp` or `/var/log` content — irrelevant to Nova's data story.
* The Docker daemon's `/var/lib/docker` tree — that is image and
  volume state, not Nova data.

Encrypt offline copies. The database contains conversation history,
user-authored memories, and per-user settings.

## Restoring from a backup

### systemd

```bash
sudo systemctl stop nova

sudo rsync -aH \
    /mnt/archive/NovaPortable-<stamp>/ \
    /mnt/fastdata/NovaPortable/

sudo systemctl start nova
journalctl -u nova -n 50 --no-pager
```

### Docker

```bash
cd /mnt/fastdata/NovaPortable
docker compose stop nova

sudo rsync -aH \
    /mnt/archive/NovaPortable-<stamp>/ \
    /mnt/fastdata/NovaPortable/

docker compose up -d
docker compose logs -f nova
```

The container picks up the restored bind-mounted directory on its
next start — no image rebuild is needed.

If you only restored the database, copy it back into the workspace's
`data/` directory:

```bash
sudo -u nova cp /mnt/archive/nova-<stamp>.db \
                /mnt/fastdata/NovaPortable/data/nova.db
```

Test the restore path on a non-production host first.

## Why stable mount paths matter

The portable workspace assumes its parent path **does not move**.
Stable mounts declared in `/etc/fstab` (`/mnt/fastdata`,
`/mnt/archive`, …) satisfy that contract: they are present at boot,
visible to every user, and survive logout.

Avoid `/run/media/<user>/<disk>` — those paths only exist while a
graphical desktop session is logged in. systemd will not wait for
them via `RequiresMountsFor=`, and they disappear on logout, taking
the live database with them. That is the easy way to corrupt
`nova.db` mid-write.

## What belongs on GitHub, and what does not

A useful mental model: the **GitHub repository holds code; the
portable workspace holds data and config**. The two never meet on the
same disk path. The split is enforced by the layout — the Git
checkout lives under `app/Nova/`, while `data/`, `config/`, `logs/`,
and `backups/` are siblings of `app/`, not children of it — so a
`git status` inside `app/Nova/` cannot see runtime files even if you
wanted it to.

**On GitHub (public, version-controlled):**

* Python source, templates, static assets, tests.
* `.env.example` (a documented set of variable names, no values).
* Documentation, including this file.
* The systemd unit example and the portable docker-compose example,
  with placeholder paths.

**Never on GitHub (private, lives in the workspace):**

* `nova.db`, `nova.db-wal`, `nova.db-shm`, and any other SQLite
  database file under `data/`.
* `data/memory-packs/`, `data/exports/`, `data/feedback/`,
  `data/settings/` — anything Nova writes at runtime.
* `config/nova.env` and any other file containing
  `NOVA_PASSWORD`, `NOVA_SECRET_KEY`, integration tokens, or OAuth
  client secrets.
* `logs/` and `backups/`.
* Operator-specific helper scripts under `scripts/` that hardcode
  hostnames, ports, or credentials.

Why the strict split:

* **Conversation history and memories are private.** They contain
  exactly the kind of personal context a user would expect Nova to
  keep on their machine, not push to a public forge.
* **Secrets in Git are practically permanent.** Even a `git rm` does
  not remove them from the history; a force-push to rewrite history
  is destructive, often blocked, and easy to miss on forks.
* **Code updates should be a `git pull`, not a migration.** Keeping
  data outside the checkout means `git pull` (and `git clean -xdf`,
  and `rm -rf app/Nova && git clone …`) never touch `nova.db`.

The repo's `.gitignore` blocks `*.db`, `*.sqlite`, `*.sqlite3`,
`.env`, `*.env`, `nova.env`, `data/`, `NovaData/`, `NovaPortable/`,
`backups/`, `exports/`, `memory-packs/`, and `logs/` defensively, so
even a stray `cp -r` into the wrong directory cannot stage runtime
data by accident. The portable workspace layout makes the defensive
rule unnecessary in normal operation — those directories are not
inside the repo to begin with — but it stays in place as a safety
net.

## Safety guarantees

These are firm boundaries of the portable workspace helper and the
deployment shapes documented above:

* `init_workspace` **never** moves or deletes data.
* `init_workspace` **never** overwrites an existing file, including
  `nova.env.example`, the workspace-level `README.md`, and any
  existing directory.
* The helper **never** clones the Nova Git repo, downloads Ollama
  models, or fetches anything from the network. Those steps remain
  explicit operator decisions.
* The helper **never** runs as root, never calls `sudo`, and never
  modifies systemd units or Docker daemons on its behalf.
* The helper exits with a clear error when the parent path is
  invalid, unwritable, or occupied by a non-directory.
* No secrets are written to Git. `config/nova.env` is created by you,
  outside the checkout, and is the one place Nova-specific secrets
  belong.
* The Docker shape does **not** mount the Docker socket, does not
  run privileged, does not mount the host root or home directory,
  and does not bundle integration tokens or Ollama models in the
  image.
* Docker isolation is **defence in depth**, not a perfect boundary.
  A container that can run arbitrary code as the in-container user
  with access to `/data` can still corrupt Nova's database. Keep
  the host patched, restrict who can run `docker`, and front the
  HTTP port with TLS or a Zero-Trust gateway when exposing it.

If you decide to abandon the portable layout, remove
`NOVA_DATA_DIR` from the env file and Nova falls back to the legacy
layout (`nova.db` next to the checkout) on the next start. Nothing
is deleted automatically.
